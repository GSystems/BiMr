Abstract

Considering tweets provided by Twitter users and the data feed provided by the eBird API, our application will be able to generate an interactive map in real-time, representing the current the tendency of birds to move from one place to another. In that way we can visualize if there is a massive migration flow.

Motivation

The idea of this project came from the fact that there are few accessible applications that allow users to see detailed information about the bird migration around the entire world. This kind of instrument can be useful in many fields, especially in ornithology, although everyone without any professional background on this field can draw conclusions after seeing a real-time updated map that illustrates this phenomena.

Twitter is an online news and social networking application where users post and read short 140-character messages called "tweets". Besides the content of the tweet, users may also make available the location, so people will know where the author was located when he posted the tweet.

eBirds API 1.1 is a public API created by Cornell University, a private and statutory Ivy League research university located in Ithaca, New York. For our particular application, it provides a real-time updated data feed concerning bird migration phenomena. For example, it can offer show pieces of information such as hotspot sightings summaries, recent notable observations at hotspots, locations, regions or of a certain species. Moreover, along with these categories of information, each one offers data about the name of sighted species (both common naming and scientific), geographical coordinates, the name of the person that made the observation etc.

Our challenge was to build a technology to collect, process and display data, all in real time. All data managed by the developed Web application must be described in RDF – based on existing knowledge models. This way, we will be able to visualize the migration of birds in a user friendly manner using a web interface updated in real time.

Our proposal

Our proposal is meant to offer a scalable solution of obtaining and analysing tweets in real time, concerning the aforementioned topic. Rather than designing a single monolithic application architecture, with all these components and services running as a single unit, we’re going to use a microservice approach, breaking down the application into independent pieces that can run on their own.

Using a microservice architecture brings many advantages, including making scaling a more easly achievable task, increased system resilience due to failure isolation, allows us to mix different libraries and frameworks and gives us the chance to extend the application for further modifications or improvements.

Architecture (Modules)

Our real time stream processing architecture is typically made of the following main components:

Data collector

A module which collects data from Twitter Public Streaming API. Once the application establishes a connection to a streaming endpoint, a feed of Tweets is piped into our processing funnel without needing to worry about reaching request quotas. We are using twitter4j, Java library to get tweets from twitter, which is then streamed to Apache Kafka. Twitter credentials are required for twitter4j to be able to pull tweets. We get these credentials by creating a new app on twitter. By all means, this does not require the user to provide his actual credentials but for each created application, Twitter provides secrets to serve in authenticating via OAuth.
We use Kafka, a reliable, high-throughput and low latency messaging system to transfer tweet data between modules. Like many publish-subscribe messaging systems, Kafka maintains feeds of messages in topics from where the second module will take tweets one by one.

Scalable message queuing with Apache Kakfa

We use Kafka, a reliable, high-throughput and low latency messaging system to transfer tweet data between modules. Like many publish-subscribe messaging systems, Kafka maintains feeds of messages in topics from where the second module will take tweets one by one.

Apache Kafka provides a scalable message queue, where producers send records to topics that are processed by consumers. In the application proposed we use we’re using message queues as the communication bus between modules. We use two queues within the application, one for queueing tweets from the Twitter and the other for filtered tweets. Using a message queue to schedule tasks makes it simple to distribute the work across multiple service instances and handle instance failure without losing work.

Kafka supports both the publish-subscribe and queueing message patterns for consumers reading records from a topic. It does this through allowing consumers to belong to a consumer group. Each message published to a topic is delivered to one consumer instance within each subscribing consumer group. If all consumer instances have the same group, this works like a traditional message queue. If all consumers have different groups, this works like publish-subscribe topics.

Data processing

Now that we have a continously fresh stream of data we can filter and process tweets. We keep only tweets which are in english and come with a place or a specified location. Then we analyse each tweet's content using Stanford NLP's TokensRegex framework, defining different patterns over text, patterns which match the intention of moving to a place. We store our tweets data using Apache Jena Fuseki which is a SPARQL server. It provides REST-style SPARQL Query, and SPARQL Update using the SPARQL protocol over HTTP.

We use it to store RDF triples and to retrieve information using SPARQL queries. An example of a SPARQL query used to select information about a migration, who wants to migrate and what contains the tweet posted, is shown below: 


PREFIX foaf: <http://xmlns.com/foaf/0.1#>
PREFIX location: <http://xmlns.com/location#>
PREFIX migration: <http://xmlns.com/migration#>
PREFIX tweet: <http://xmlns.com/tweet#>
PREFIX xsd: <http://www.w3.org/2001/XMLSchema#>
SELECT DISTINCT ?migration ?lat_from ?long_from ?country_from ?country_code_from ?city_from ?date ?lat_to ?long_to
?country_to ?city_to ?country_code_to ?lat_from ?tweet_id ?tweet_text ?tweet_language ?person_account ?person_name
WHERE {
  ?migration     migration:from ?location_from.
  ?location_from location:latitude ?lat_from.

  ?location_from  location:longitude ?long_from.
  OPTIONAL { ?location_from  location:country ?country_from}.
  OPTIONAL { ?location_from  location:city ?city_from } .
  OPTIONAL { ?location_from  location:country_code ?country_code_from. }

  ?migration     migration:to ?location_to.
  ?location_to   location:latitude ?lat_to;
  location:longitude ?long_to.
          OPTIONAL { ?location_to  location:country ?country_to. }
          OPTIONAL { ?location_to    location:city ?city_to . }
          OPTIONAL { ?location_to  location:country_code ?country_code_to. }

  ?migration     migration:date ?date;
                 migration:person ?person_account .

  ?migration     migration:tweetId ?tweet_id.
  ?tweet         tweet:id ?tweet_id;
                 tweet:text ?tweet_text.
      OPTIONAL { ?tweet tweet:language ?tweet_language . }
  ?foaf          foaf:accountName ?person_account;
                 foaf:name ?person_name.
FILTER( ?date<="2020-02-09T21:42:15"^^xsd:dateTime && ?date >="1994-03-28T00:00:00"^^xsd:dateTime)
}
